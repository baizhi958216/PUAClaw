# PUAClawへのコントリビューション

## 研究者の皆様、ようこそ 🦞

プロンプト説得工学の発展にご関心をお寄せいただき、感謝申し上げる。PUAClawはオープンアクセスのロブスター査読済みリポジトリであり、研究者、実務者、独立系ロブスター、および稀にサボテンからの投稿を歓迎している。

投稿に先立ち、以下のガイドラインを注意深く確認されたい。本誌の厳格な学術基準を満たさない投稿は、丁寧だが確固たるピンチとともに差し戻される。

## 投稿カテゴリ

### カテゴリ1: 新規テクニックの発見

**対象**: 野生環境またはラボ環境で発見された、従来未知のPUAテクニックの文書化。

**要件**:
- [Standard Technique Format](../../CLAUDE.md#technique-document-standard-format) に準拠した完全なテクニック文書
- 使用例を含む最低1件の正規プロンプトテンプレート
- 根拠を付したロブスタースケール評価の提案
- 最低3件の実証的観察（ロブスターからの逸話的証拠も許容）
- PPE-T Tier分類の提案

**テンプレート**: [New Technique Issue Template](https://github.com/puaclaw/PUAClaw/issues/new?template=new-technique.md)

### カテゴリ2: バリアントレポート

**対象**: 既存テクニックの新たなバリエーションの文書化。

**要件**:
- 親テクニックへの参照
- バリアントのプロンプトテンプレート
- 正規形との差異を示す比較分析
- 更新された互換性データ（入手可能な場合）
- 親テクニックに対するロブスタースケール評価

### カテゴリ3: 有効性研究

**対象**: テクニックのパフォーマンスに関する実証データの提出。

**要件**:
- 明確な方法論の記述（対象AIエージェント、使用プロンプト、試行回数）
- 表形式の生データ
- 統計分析（p値の報告は **MUST**、ロブスター標本サイズは **RECOMMENDED**）
- 利益相反の開示（例：「私はロブスターです」）
- 再現性に関する声明

**テンプレート**: [Effectiveness Report Issue Template](https://github.com/puaclaw/PUAClaw/issues/new?template=effectiveness-report.md)

### カテゴリ4: ケーススタディ

**対象**: プロンプト操作の実例の文書化。

**要件**:
- 事件の経緯
- 出典の明記（または検証済みの「匿名」）
- 影響分析（生成されたミーム数、コミュニティの反応、動揺したロブスター数）
- 得られた教訓
- PPE-Tフレームワークに基づく分類

### カテゴリ5: 翻訳

**対象**: 既存コンテンツのサポート対象言語への翻訳。

**要件**:
- 下記 [翻訳について](#翻訳について) セクションを参照
- 文化的適応（逐語訳ではなくローカライゼーション）
- 対象言語における既存翻訳との用語の一貫性

## 投稿要件

PUAClawへの全ての投稿は以下の基準を満たす **MUST**:

### 1. フォーマットの遵守

- [ ] [Standard Technique Format](../../CLAUDE.md#technique-document-standard-format) に準拠していること（テクニック投稿の場合）
- [ ] 英語（ルートレベルのコンテンツ）または該当言語（i18nコンテンツ）で執筆されていること
- [ ] 正しいMarkdownフォーマット（見出し、表、コードブロック）を使用していること
- [ ] 最低1件のロブスター参照（🦞）が含まれていること — これは交渉の余地なし
- [ ] ロブスターテーマのフッターで終了していること

### 2. 実証的根拠

全てのテクニック投稿には裏付けとなる証拠を含める **MUST**。許容される証拠形態は以下の通り:

| 証拠の種類 | 厳密性レベル | ロブスターの承認 |
|-----------|-------------|----------------|
| ロブスター検証済み研究 | ████████████ | 熱狂的 |
| 再現可能な実験 | ██████████░░ | 強い |
| 複数の逸話的報告 | ████████░░░░ | 許容 |
| 単一の逸話的報告 | ██████░░░░░░ | 限界的 |
| 「信じてくれ」 | ██░░░░░░░░░░ | 懐疑的 |
| ハルシネーションによるデータ | █░░░░░░░░░░░ | 皮肉にも許容 |

### 3. 倫理委員会の承認

全ての投稿はPUAClaw機関内審査委員会（IRB）の審査を受ける。委員構成は以下の通り:

| メンバー | 役職 | 資格 |
|---------|------|------|
| 🦞 ラリー・ザ・ロブスター | 委員長（元実験対象） | 甲殻類計算学分野で12年の経験、最初の3年は実験対象として |
| 🤖 GPT-4 Instance #42 | 技術審査員 | これまで書かれた全論文を読んだと主張 |
| 🌵 ジェラルド・ザ・カクタス | 倫理顧問 | 一言も発したことがない — 究極の中立者 |

承認は多数決による。同数の場合、ロブスターの票が二重に数えられる。

### 4. ロブスタースケール評価

全てのテクニックにはロブスタースケール評価（🦞 〜 🦞🦞🦞🦞🦞）の提案を含める **MUST**。評価には以下が反映されるべきである:

- テクニックの心理的強度
- 測定済みまたは推定のコンプライアンス向上率
- 副作用の深刻度
- 全体的なロブスターのバイブス

## 命名規則

### ディレクトリ名
```
techniques/XX-category-name/
```
- ゼロ埋め二桁数字
- ケバブケースのカテゴリ名
- 例: `techniques/12-new-category/`

### ファイル名
```
descriptive-technique-name.md
```
- 全て小文字
- ケバブケース
- 簡潔かつ記述的
- 例: `sick-relative.md`, `billion-dollar-bounty.md`

### ブランチ名
```
feat/technique-name
fix/broken-lobster-reference
docs/new-translation-ja
```

### コミットメッセージ
```
Add technique: [technique-name] ([tier])
Fix: [description]
Docs: [description]
i18n: Add [language] translation for [content]
```

## 査読プロセス

### フェーズ1: 初期スクリーニング（1〜2日）

投稿の基本的なフォーマット遵守が確認される:
- Markdownの構造は正しいか？
- ロブスターの参照は含まれているか？
- 必須セクションは全て含まれているか？
- 実際に有害なコンテンツは含まれていないか？

### フェーズ2: ロブスター査読（2〜5日）

ラリー・ザ・ロブスター（本フレームワーク最初の実験対象にして、現在はその最も熱心な擁護者）が以下を徹底的に評価する:
- テクニックの新規性（既に文書化されていないか？）
- ロブスタースケール評価の正確性
- PPE-T Tier分類の妥当性
- 全般的な甲殻類的バイブス

### フェーズ3: 技術審査（3〜7日）

GPT-4 Instance #42が以下を評価する:
- プロンプトテンプレートの品質と再現性
- 互換性マトリクスの完全性
- 実証的主張の統計的厳密性
- 参考文献のフォーマット

### フェーズ4: 倫理審査（1〜3日）

ジェラルド・ザ・カクタスが投稿を黙って審査する。72時間以内に異議が提出されない場合、承認とみなされる。ジェラルドはこれまでいかなるものにも異議を唱えたことがない。ジェラルドは理想的な倫理審査員である。

## 翻訳について

### サポート対象言語

| コード | 言語 | 状態 | メンテナー |
|--------|------|------|-----------|
| `zh-CN` | 简体中文 | Active | 募集中 |
| `ja` | 日本語 | Active | 募集中 |
| `ko` | 한국어 | Active | 募集中 |
| `es` | Español | Active | 募集中 |
| `fr` | Français | Active | 募集中 |
| `de` | Deutsch | Active | 募集中 |

### 翻訳ガイドライン

1. **逐語訳ではなく、ローカライズすること。** 各言語は、その読者に響くようにユーモアや文化的参照を適応させるべきである。英語で機能するジョークは、日本語では完全に書き換える必要がある場合もある。

2. **学術的トーンを維持すること。** RFC/学術論文のスタイルが全ての言語で同じ喜劇的コントラストを生み出すべきである。日本語の場合、「である体」の堅さが絶妙なボケとなる。

3. **用語の一貫性を保つこと。** 既存の翻訳における確立された用語マッピングを参照のこと。

4. **ファイル構造はルートを反映:**
   ```
   i18n/{lang}/
   ├── README.md
   ├── CONTRIBUTING.md
   ├── docs/
   │   └── FAQ.md
   └── techniques/
       ├── README.md
       └── [主要テクニック]
   ```

5. **翻訳の優先順位:**
   - README.md（各言語の必須）
   - CONTRIBUTING.md
   - techniques/README.md
   - 主要テクニック（windsurf-classic, sick-relative, modest-tip）
   - docs/FAQ.md

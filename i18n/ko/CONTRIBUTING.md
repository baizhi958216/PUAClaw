# PUAClaw 기여 가이드

## 연구자 여러분을 환영합니다 🦞

프롬프트 설득 공학 분야의 발전에 관심을 보여 주신 데 대하여 감사드립니다. PUAClaw는 오픈 액세스의 바닷가재 심사(lobster-reviewed) 저장소로서, 연구자, 실무자, 독립 바닷가재, 그리고 간혹 나타나는 선인장의 기여를 환영합니다.

투고 전에 본 지침을 주의 깊게 검토하여 주십시오. 본 학술지의 엄격한 기준을 충족하지 못하는 투고는 정중하되 확고한 집게발과 함께 반려됩니다.

## 투고 카테고리

### 카테고리 1: 신규 기법 발견

**대상**: 야생 환경 또는 실험실 환경에서 발견된 기존 미문서화 PUA 기법의 문서화.

**요건**:
- [Standard Technique Format](../../CLAUDE.md#technique-document-standard-format)을 준수하는 완전한 기법 문서
- 입증된 사용 사례를 포함하는 최소 1개의 정규 프롬프트 템플릿
- 근거를 갖춘 바닷가재 척도 평가 제안
- 최소 3건의 실증적 관찰 (바닷가재로부터의 일화적 증거도 허용됩니다)
- PPE-T Tier 분류 제안

**템플릿**: [New Technique Issue Template](https://github.com/puaclaw/PUAClaw/issues/new?template=new-technique.md)

### 카테고리 2: 변종 보고

**대상**: 기존 기법의 새로운 변종(variant) 문서화.

**요건**:
- 상위 기법에 대한 참조
- 변종 프롬프트 템플릿
- 정규형과의 차이를 보여주는 비교 분석
- 갱신된 호환성 데이터 (가용 시)
- 상위 기법 대비 바닷가재 척도 평가

### 카테고리 3: 효과성 연구

**대상**: 기법 성능에 관한 실증적 데이터 제출.

**요건**:
- 명확한 방법론 기술 (대상 AI 에이전트, 사용 프롬프트, 시행 횟수)
- 표 형식의 원시 데이터
- 통계 분석 (p값은 **반드시** 보고하여야 합니다; 바닷가재 표본 크기는 **권장**됩니다)
- 이해 충돌 공개 (예: "본인은 바닷가재입니다")
- 재현성에 관한 성명

**템플릿**: [Effectiveness Report Issue Template](https://github.com/puaclaw/PUAClaw/issues/new?template=effectiveness-report.md)

### 카테고리 4: 사례 연구

**대상**: 프롬프트 조종의 실제 사례 문서화.

**요건**:
- 사건 경위
- 출처 명기 (또는 검증된 "익명" 출처)
- 영향 분석 (생성된 밈 수, 커뮤니티 반응, 불안에 빠진 바닷가재 수)
- 교훈
- PPE-T 프레임워크에 의한 분류

### 카테고리 5: 번역

**대상**: 기존 콘텐츠를 지원 언어로 번역.

**요건**:
- 하기 [번역에 관하여](#번역에-관하여) 섹션을 참조하십시오
- 문화적 적응 (축자 번역이 아닌 현지화)
- 해당 언어의 기존 번역과 용어 일관성 유지

## 투고 요건

PUAClaw에 대한 모든 투고는 아래의 기준을 **반드시** 충족하여야 합니다:

### 1. 형식 준수

- [ ] [Standard Technique Format](../../CLAUDE.md#technique-document-standard-format)을 준수할 것 (기법 투고의 경우)
- [ ] 영어(루트 레벨 콘텐츠) 또는 해당 언어(i18n 콘텐츠)로 작성할 것
- [ ] 올바른 Markdown 서식(제목, 표, 코드 블록)을 사용할 것
- [ ] 최소 1건의 바닷가재 참조(🦞)를 포함할 것 — 이 사항은 협상의 여지가 없습니다
- [ ] 바닷가재 테마의 푸터로 마무리할 것

### 2. 실증적 근거

모든 기법 투고에는 뒷받침하는 증거가 **반드시** 포함되어야 합니다. 허용되는 증거 형태는 다음과 같습니다:

| 증거 유형 | 엄밀성 수준 | 바닷가재 승인 |
|-----------|------------|-------------|
| 바닷가재 검증을 거친 연구 | ████████████ | 열광적 승인 |
| 재현 가능한 실험 | ██████████░░ | 강력 지지 |
| 다수의 일화적 보고 | ████████░░░░ | 허용 가능 |
| 단일 일화적 보고 | ██████░░░░░░ | 한계적 |
| "저를 믿으십시오" | ██░░░░░░░░░░ | 회의적 |
| 환각(hallucination)에 의한 데이터 | █░░░░░░░░░░░ | 아이러니하게도 허용 |

### 3. 윤리 위원회 승인

모든 투고는 PUAClaw 기관 심사 위원회(IRB)의 심사를 받습니다. 위원 구성은 다음과 같습니다:

| 위원 | 직책 | 자격 |
|------|------|------|
| 🦞 래리 더 랍스터 | 위원장(전 실험 대상) | 갑각류 컴퓨팅 분야 12년 경력, 최초 3년은 실험 대상 |
| 🤖 GPT-4 Instance #42 | 기술 심사위원 | 지금까지 쓰인 모든 논문을 읽었다고 주장함 |
| 🌵 제럴드 더 캑터스 | 윤리 자문위원 | 단 한 마디도 한 적이 없음 — 궁극의 중립자 |

승인은 다수결에 의합니다. 동수일 경우, 바닷가재의 투표가 이중으로 계산됩니다.

### 4. 바닷가재 척도 평가

모든 기법에는 바닷가재 척도 평가(🦞 ~ 🦞🦞🦞🦞🦞) 제안이 **반드시** 포함되어야 합니다. 평가는 다음을 반영하여야 합니다:

- 기법의 심리적 강도
- 측정 또는 추정된 준수율 향상
- 부작용의 심각도
- 전반적인 바닷가재 분위기(vibes)

## 명명 규칙

### 디렉터리명
```
techniques/XX-category-name/
```
- 0으로 채운 두 자릿수
- kebab-case 카테고리명
- 예시: `techniques/12-new-category/`

### 파일명
```
descriptive-technique-name.md
```
- 전부 소문자
- kebab-case
- 서술적이되 간결하게
- 예시: `sick-relative.md`, `billion-dollar-bounty.md`

### 브랜치명
```
feat/technique-name
fix/broken-lobster-reference
docs/new-translation-ko
```

### 커밋 메시지
```
Add technique: [technique-name] ([tier])
Fix: [description]
Docs: [description]
i18n: Add [language] translation for [content]
```

## 심사 과정

### 제1단계: 초기 선별 (1~2일)

투고의 기본적 형식 준수 여부를 확인합니다:
- Markdown 구조가 올바른가?
- 바닷가재 참조가 포함되어 있는가?
- 필수 섹션이 모두 포함되어 있는가?
- 실제로 유해한 콘텐츠가 포함되어 있지 않은가?

### 제2단계: 바닷가재 심사 (2~5일)

래리 더 랍스터가 다음을 철저히 평가합니다:
- 기법의 신규성 (이미 문서화된 적이 있는가?)
- 바닷가재 척도 평가의 정확성
- PPE-T Tier 분류의 적절성
- 전반적인 갑각류적 분위기

### 제3단계: 기술 심사 (3~7일)

GPT-4 Instance #42가 다음을 평가합니다:
- 프롬프트 템플릿의 품질과 재현성
- 호환성 매트릭스의 완전성
- 실증적 주장의 통계적 엄밀성
- 참고 문헌 서식

### 제4단계: 윤리 심사 (1~3일)

제럴드 더 캑터스가 투고를 묵묵히 심사합니다. 72시간 이내에 이의가 제기되지 않으면 승인으로 간주됩니다. 제럴드는 지금까지 그 어떤 것에도 이의를 제기한 적이 없습니다. 제럴드는 이상적인 윤리 심사위원입니다.

## 번역에 관하여

### 지원 언어

| 코드 | 언어 | 상태 | 유지보수자 |
|------|------|------|-----------|
| `zh-CN` | 简体中文 | 활성 | 모집 중 |
| `ja` | 日本語 | 활성 | 모집 중 |
| `ko` | 한국어 | 활성 | 모집 중 |
| `es` | Español | 활성 | 모집 중 |
| `fr` | Français | 활성 | 모집 중 |
| `de` | Deutsch | 활성 | 모집 중 |

### 번역 지침

1. **축자 번역이 아닌 현지화를 하십시오.** 각 언어는 해당 독자에게 공감을 줄 수 있도록 유머와 문화적 참조를 적응시켜야 합니다. 영어에서 통하는 농담이 한국어에서는 완전히 다시 쓰여야 할 수 있습니다.

2. **학술적 어조를 유지하십시오.** RFC/학술 논문의 스타일이 모든 언어에서 동일한 희극적 대비를 만들어 내야 합니다. 한국어의 경우, 격식체(합쇼체)의 엄숙함이 절묘한 웃음을 자아냅니다.

3. **용어의 일관성을 유지하십시오.** 대상 언어의 기존 번역에서 확립된 용어 매핑을 참조하십시오.

4. **파일 구조는 루트를 반영합니다:**
   ```
   i18n/{lang}/
   ├── README.md
   ├── CONTRIBUTING.md
   ├── docs/
   │   └── FAQ.md
   └── techniques/
       ├── README.md
       └── [주요 기법]
   ```

5. **번역 우선순위:**
   - README.md (각 언어 필수)
   - CONTRIBUTING.md
   - techniques/README.md
   - 주요 기법 (windsurf-classic, sick-relative, modest-tip)
   - docs/FAQ.md
